# 혼자 만드는 멀티채널 콘텐츠 자동화, 어떻게 “시스템”으로 만들 것인가

  콘텐츠를 만드는 게 아니라, 콘텐츠를 만드는 공장을 만든 이야기

  콘텐츠 자동화를 처음 시작할 때 대부분 이렇게 생각합니다.
  “좋은 프롬프트 하나 만들면 되지 않을까?”

  저도 그랬습니다.
  하지만 조금만 운영해보면 금방 한계가 옵니다.

  - 채널이 늘어날수록 품질이 흔들리고
  - 발행이 쌓일수록 운영이 복잡해지고
  - 잘 되던 것도 어느 날 갑자기 망가집니다.

  결국 문제는 콘텐츠 생성이 아니라, 콘텐츠 운영 시스템 설계였습니다.
  제가 ContentForge를 구성하면서 얻은 방법론을 공유합니다.

  ———

  ## 1. “AI가 글 써주는 앱”이 아니라, “파이프라인 운영 플랫폼”으로 정의하기

  처음부터 제품 정체성을 이렇게 잡았습니다.

  - 입력: 소재(아이디어, RSS, 트렌드, 북마크)
  - 처리: 수집 → 선별 → 변환 → 검증 → 발행 → 분석
  - 출력: 채널별 결과물 + 성과 데이터
  - 피드백: Build-Measure-Learn(BML) 루프

  핵심은 단순합니다.
  생성(Generate)보다 운영(Operate)을 먼저 설계해야 합니다.

  ———

  ## 2. 기능보다 경계(Boundary)를 먼저 자른다

  프로젝트는 monorepo지만, 패키지 경계를 강하게 잡았습니다.

  - core: 타입, Result 패턴, 로깅
  - collectors: 소재 수집
  - agents: 의사결정/작업 실행
  - pipelines: 채널 변환
  - publishers: 채널 발행 어댑터
  - analytics: 성과 수집/리포트
  - video, web, cli: 인터페이스와 실행 도구

  이 구조의 장점은 “좋은 아키텍처”라서가 아닙니다.
  문제가 생겼을 때 책임 위치가 명확해진다는 점입니다.

  ———

  ## 3. 예외 처리 철학을 통일한다: throw보다 Result

  자동화 시스템은 실패가 전제입니다.
  API 제한, 네트워크 오류, 스키마 불일치, 정책 위반은 매일 발생합니다.

  그래서 비즈니스 로직에서는 가능한 한
  Result<T, E> 패턴으로 성공/실패를 명시적으로 다뤘습니다.

  이렇게 하면:

  - 실패 경로를 숨기지 않게 되고
  - 재시도/보류/수동개입 정책을 코드로 일관되게 넣을 수 있습니다.

  ———

  ## 4. “완벽한 자동화” 대신 “개입 가능한 자동화”를 만든다

  현실적으로 중요한 건 자동화율 100%가 아닙니다.
  사람이 언제, 어떻게 개입할 수 있는가입니다.

  그래서 운영 화면에서 반드시 제공해야 할 기능을 정했습니다.

  - 승인/보류/수정요청
  - 파이프라인 강제 라우팅
  - 실패 건 재시도
  - 수동 재배포
  - 드래프트 편집 후 승인

  자동화는 속도를 만들고,
  사람의 개입은 브랜드와 신뢰를 지킵니다.

  ———

  ## 5. 문서와 코드의 불일치를 “운영 리스크”로 취급한다

  이 프로젝트에서 가장 크게 배운 점입니다.
  문서-코드 불일치는 단순 불편이 아니라 사고의 시작점입니다.

  예를 들어:

  - 문서엔 10개 에이전트인데 코드엔 7개만 구현
  - 타입 채널명과 DB enum 채널명이 다름

  이런 상태는 팀 합류자, 운영자, 자동화 툴 모두를 헷갈리게 만듭니다.
  그래서 문서는 설명서가 아니라 실행 계약서로 다뤄야 합니다.

  ———

  ## 6. 단계별 로드맵은 “기술 난이도”가 아니라 “운영 위험” 기준으로 짠다

  제가 적용한 우선순위는 이렇습니다.

  1. 스키마/명세 정합성 복구
  2. 실행 경로 통합(오케스트레이터 중심)
  3. 검증 강제(스키마, 정책, 재시도)
  4. UI 고도화

  예쁜 화면보다 먼저 해야 할 건
  망가지지 않는 흐름입니다.

  ———

  ## 7. 최종적으로 남는 건 프롬프트가 아니라 시스템 습관

  콘텐츠 자동화는 결국 사람의 운영 습관을 코드화하는 과정입니다.

  - 어떤 소재를 고를지
  - 어떤 채널로 보낼지
  - 어떤 기준에서 승인할지
  - 어떤 실패를 재시도할지
  - 어떤 지표를 보고 전략을 바꿀지

  이 질문들에 대한 팀의 답이 아키텍처가 됩니다.
